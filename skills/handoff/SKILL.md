---
name: handoff
description: Use when the user explicitly invokes handoff to preserve context for another agent or asks to continue from a handoff id such as #task-id.
---

一次性事实接力快照。旧 agent 的想法不是事实来源；handoff 只迁移用户确认、一手证据和影响继续工作的未验证缺口。

## 模式选择

| 用户输入 | 模式 |
| --- | --- |
| `/handoff`、`$handoff`、不带 `#id` 的 handoff 请求 | 创建快照 |
| `/handoff 继续...#<id>` 或明确要求继续 `#<id>` | 继续快照 |
| 说继续但没有 `#id` | 要求用户补 ID，不要猜 |

只能手动触发；不要隐式使用

## 创建快照

1. 自动生成 ID：从当前已确认任务名提取短文件名，去掉 `/`、`..` 等路径字符；无清晰任务时用 `<YYYY-MM-DD-HHMM>-handoff`；重名追加 `-2`。不要要求用户手动提供创建 ID。
2. 写入当前仓库的 `docs/handoffs/<id>.md`，目录不存在就创建。
3. 如果用户给了不含 `#id` 且不是继续请求的参数，把它当作下一次会话的主要目标提示，仍然是创建模式。

文档按这个结构写，空节省略，但“目标”和“下一步”必须有：

```md
# <handoff-id>

## 目标
- [用户确认] ...

## 已确认约束
- [用户确认] ...

## 一手证据
- [文件] path/to/file: 说明它证明什么
- [命令] `command`: 只写关键结果

## 已完成
- [证据] ...

## 未完成 / 下一步
- [事实] 下一位 agent 应先做的最小动作

## 未验证
- [未验证] 只写会影响继续工作的缺口

## 不要做
- [用户确认] ...

## 建议使用的 skills
- skill-name：为什么下一位 agent 需要它
```

## 信息取舍

| 信息类型 | 写法 |
| --- | --- |
| 用户明确确认 | 写入，标 `[用户确认]` |
| 文件、命令、测试、commit、issue、PRD、plan、ADR、diff | 引用路径、URL 或命令结果，标 `[文件]`、`[命令]` 等 |
| 旧 agent 的猜测 | 默认删除；只有用户明确要求“不要采纳这个猜测”时，写入“不要做” |
| 未验证状态 | 只写事实缺口，例如“未跑测试”；不要补可能原因 |

不要重复已经存在于其他产物里的内容，例如 PRD、plan、ADR、issue、commit 或 diff。对这些内容只给出路径或 URL 引用，不要整段抄写。

移除敏感信息，例如 API key、密码或个人隐私信息。

## 继续快照

继续模式必须有 `#<id>`。读取 `docs/handoffs/<id>.md`；找不到时先在 `docs/handoffs/` 按文件名搜索，仍找不到就说明缺少 handoff。

读取后先输出理解校准，不要直接开始实现：

```md
## Handoff 理解校准

- 我确认的事实：
- 未完成事项：
- 需要重新验证：
- 建议用户补充：
```

等用户确认或修正后再继续。

## 常见错误

| 错误 | 正确做法 |
| --- | --- |
| 写成聊天记录摘要 | 写成可执行的事实接力单 |
| 把旧 agent 推理写成事实 | 删除；没有用户确认或一手证据就不写 |
| 为完整而复制 plan/diff | 只引用路径或 URL |
| 列一串通用 skills | 只列下一步真正需要的 1-3 个 |
| 说继续但没有 `#id` | 要求用户补 ID，不要猜测要读哪个 handoff |
